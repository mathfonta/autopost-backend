# Story 3.1 — Endpoints de Submissão de Foto e Histórico

**Status:** Done  
**Epic:** 3 — API de Conteúdo  
**Story:** 3.1  
**Estimativa:** 5 pontos  
**Depende de:** 1.2 (Auth JWT), 1.4 (Pipeline Celery), 2.3 (Designer — upload R2)

---

## Descrição

Como cliente autenticado, preciso enviar uma foto via API para que o pipeline de agentes seja disparado automaticamente, e consultar o status e resultado do processamento.

---

## Critérios de Aceitação

- [ ] AC1: `POST /content-requests` aceita `multipart/form-data` com campo `photo` (arquivo de imagem) — faz upload para R2 e cria `ContentRequest` com status `pending`
- [ ] AC2: Após criação, `start_content_pipeline(request_id)` é chamado imediatamente — retorna `{id, status, celery_task_id}`
- [ ] AC3: Formatos aceitos: JPEG, PNG, WEBP — tamanho máximo 20MB — retorna 422 se inválido
- [ ] AC4: `GET /content-requests/{id}` retorna o objeto completo com `status`, `analysis_result`, `copy_result`, `design_result`, `publish_result` (campos nulos se ainda não processados)
- [ ] AC5: `GET /content-requests` retorna lista paginada (`page`, `page_size`, padrão 20) dos requests do cliente autenticado (multi-tenancy via `client_id`)
- [ ] AC6: Endpoint protegido por JWT — `401` se token ausente/inválido, `403` se tentar acessar request de outro cliente
- [ ] AC7: Testes cobrindo upload, pipeline trigger, listagem e controle de acesso

## Dev Notes

- Upload deve usar `app/core/storage.py:upload_to_r2()` — já implementado na Story 2.3
- Chave R2: `uploads/{client_id}/{uuid4}.jpg` — sufixo sempre `.jpg` após conversão
- `source_channel` padrão: `"app"` (campo já existe no modelo)
- `client_id` vem do JWT: `current_user.id` — padrão já estabelecido nos endpoints de auth (Story 1.2)
- Multi-tenancy: `ContentRequest` já tem `TenantMixin` com `client_id` — usar no filtro do `GET /content-requests`
- Para o GET /{id}: retornar 404 se não encontrado, 403 se `client_id` não bate
- Pipeline é fire-and-forget: não aguardar resultado, retornar imediatamente com `celery_task_id`
- Schema de response do detalhe deve incluir `design_result.processed_photo_url` para o frontend mostrar preview

## Riscos

| Risco | Probabilidade | Mitigação |
|-------|--------------|-----------|
| Upload de arquivos grandes (>20MB) causa timeout no Railway | Média | Limite de 20MB já no AC3; validar antes de fazer upload para R2 |
| R2 indisponível durante upload | Baixa | `upload_to_r2` deve levantar exceção — retornar 503 com mensagem amigável |
| Pipeline Celery não conectado no ambiente de teste | Média | Mockar `start_content_pipeline` nos testes — já previsto no AC7 |
