# Story 3.2 — Endpoints de Aprovação e Rejeição

**Status:** Done  
**Epic:** 3 — API de Conteúdo  
**Story:** 3.2  
**Estimativa:** 3 pontos  
**Depende de:** 3.1 (ContentRequest criado), 2.4 (Agente Publicador)

---

## Descrição

Como cliente autenticado, preciso aprovar ou rejeitar um post que passou pelo pipeline de agentes, para que o Agente Publicador seja disparado (aprovação) ou o request seja cancelado (rejeição).

---

## Critérios de Aceitação

- [ ] AC1: `POST /content-requests/{id}/approve` — valida que `status == awaiting_approval`, muda para `approved` e dispara `publish_post.delay(request_id)`
- [ ] AC2: Retorna `{id, status: "publishing"}` imediatamente após disparar a task (fire-and-forget)
- [ ] AC3: `POST /content-requests/{id}/reject` — aceita body opcional `{"reason": "texto"}`, muda status para `failed`, salva motivo em `error_message`
- [ ] AC4: Aprovação com status diferente de `awaiting_approval` retorna `409 Conflict` com mensagem descritiva
- [ ] AC5: Endpoints protegidos por JWT — `403` se `client_id` do token não bate com o da request
- [ ] AC6: Testes cobrindo aprovação, rejeição, conflito de status e controle de acesso

## Dev Notes

- `publish_post` é uma Celery task importada de `app/tasks/pipeline` — chamar com `.delay(request_id)` (não `.apply_async` pois não precisa de countdown aqui)
- Status após aprovação: atualizar para `approved` no banco e depois disparar task — a task muda para `publishing` quando começa
- Importar `publish_post` dentro da função para evitar circular import (mesmo padrão dos outros endpoints)
- `reject` não tem validação de status — qualquer request pode ser rejeitado (inclusive failed, para limpar histórico)
- `error_message` para rejeição: `f"Rejeitado pelo cliente: {reason}"` se reason fornecido, senão `"Rejeitado pelo cliente"`

## Riscos

| Risco | Probabilidade | Mitigação |
|-------|--------------|-----------|
| Double-submit: cliente clica em aprovar duas vezes | Média | Verificação de status no início — segundo request recebe 409 |
| publish_post.delay falha silenciosamente (Celery broker offline) | Baixa | Mockar no teste; em produção, Celery worker monitora fila |
| Status `approved` no banco mas task não enfileirada (crash entre as duas operações) | Baixa | Transação DB + enfileiramento fora da transação é aceitável no MVP |
